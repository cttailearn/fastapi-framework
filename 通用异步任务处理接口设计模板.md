**版本：1.0**

## 1. 设计目标

本接口模板旨在为各类需要**异步处理**的后端服务提供一个统一、稳定、可扩展的 API 设计规范。适用于但不限于以下场景：

- 内容审核（文本、图像、视频、音频）

- 数据处理（文件转换、数据分析）

- AI 模型推理（语音识别、图像生成）

- 批量任务处理（邮件发送、报告生成）



**核心设计原则**：

- **统一入口**：所有任务通过同一个接口提交，通过 `type` 区分任务类型。

- **异步处理**：避免客户端长时间等待，提交后立即返回任务 ID，客户端轮询或通过回调获取结果。

- **状态管理**：提供完整的任务生命周期管理（提交、查询、取消、删除）。

- **可扩展性**：数据结构预留扩展字段，新增任务类型或字段无需修改接口定义。

- **稳定性**：接口版本化，保证向后兼容。

## 2. 接口概览

|接口|方法|路径|功能|
|-|-|-|-|
|提交任务|`POST`|`/v1/tasks`|创建一个新的异步任务|
|查询任务|`GET`|`/v1/tasks/{taskId}`|获取任务进度和结果|
|取消任务|`POST`|`/v1/tasks/{taskId}/cancel`|取消正在处理的任务|
|删除任务|`DELETE`|`/v1/tasks/{taskId}`|删除任务及其关联数据|

## 3. 接口详情

### 3.1 提交任务

**请求**

- **方法**：`POST`

- **路径**：`/v1/tasks`

- **Content-Type**：`multipart/form-data`（支持文件上传和 JSON 混合）或 `application/json`（纯文本参数）

**请求参数**

|字段名|类型|必填|描述|
|-|-|-|-|
|`type`|string|是|任务类型，如 `"text"`、`"image"`、`"video"`，由业务方定义|
|`data`|object|否|任务数据，根据 `type` 不同而不同，可包含字段如 `content`、`url` 等|
|`file`|file|否|上传的文件，与 `data` 二选一或结合使用|
|`callback`|string|否|任务完成后的回调 URL（HTTP POST），服务端将任务结果推送到该地址|
|`config`|object|否|任务配置参数，如处理超时、优先级等，预留扩展|

**响应**

- **状态码**：`201 Created`

- **Content-Type**：`application/json`

```JSON
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "tsk_1234567890",
    "type": "video",
    "status": "pending",
    "createdAt": "2025-03-15T10:00:00Z"
  }
}
```

- `code`：业务状态码，`0` 表示成功，非零表示错误。

- `message`：状态描述。

- `data`：任务对象

### 3.2 查询任务

**请求**

- **方法**：`GET`

- **路径**：`/v1/tasks/{taskId}`

**路径参数**

- `taskId`：任务唯一标识，由提交任务接口返回。

**响应**

- **状态码**：`200 OK`

- **Content-Type**：`application/json`

```JSON
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "tsk_1234567890",
    "type": "video",
    "status": "processing",          // 任务状态：处理中
    "progress": 45,                  // 进度百分比，表示已完成45%
    "createdAt": "2025-03-15T10:00:00Z",
    "updatedAt": "2025-03-15T10:01:30Z",
    "estimatedRemainingSeconds": 120 // 可选：预计剩余秒数
  }
}
```

```JSON
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "tsk_1234567890",
    "type": "video",
    "status": "completed",
    "progress": 100,
    "createdAt": "2025-03-15T10:00:00Z",
    "updatedAt": "2025-03-15T10:01:30Z",
    "completedAt": "2025-03-15T10:01:30Z",
    "result": {
      "suggestion": "block",
      "details": { ... }
    }
  }
}
```

- 若任务不存在，返回 `404 Not Found`，错误信息见 4.2 节。

---

### 3.3 取消任务

**请求**

- **方法**：`POST`

- **路径**：`/v1/tasks/{taskId}/cancel`

**路径参数**

- `taskId`：任务 ID。

**响应**

- **状态码**：`200 OK`

- **Content-Type**：`application/json`

```JSON
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "tsk_1234567890",
    "status": "cancelled",
    "updatedAt": "2025-03-15T10:02:00Z"
  }
}
```

- 若任务无法取消（如已完成或不存在），返回 `400 Bad Request` 或 `404 Not Found`。

---

### 3.4 删除任务

**请求**

- **方法**：`DELETE`

- **路径**：`/v1/tasks/{taskId}`

**路径参数**

- `taskId`：任务 ID。

**响应**

- **状态码**：`204 No Content`（成功，无返回体）

- 若任务不存在或不允许删除（如处理中），返回对应的错误码。

## 4. 数据结构定义

### 4.1 任务对象（Task）

|字段名|类型|必填|描述|
|-|-|-|-|
|`taskId`|string|是|任务唯一 ID|
|`type`|string|是|任务类型，由业务定义|
|`status`|string|是|任务状态：`pending`（排队中）、`processing`（处理中）、`completed`（完成）、`failed`（失败）、`cancelled`（已取消）|
|`createdAt`|string|是|创建时间（ISO 8601）|
|`updatedAt`|string|是|最后更新时间|
|`completedAt`|string|否|完成时间（仅当 `status` 为 `completed` 或 `failed` 时存在）|
|`result`|object|否|任务结果，当 `status=completed` 时存在|
|`error`|object|否|错误信息，当 `status=failed` 时存在，包含 `code` 和 `message`|

### 4.2 统一响应格式

所有接口返回的 JSON 结构遵循以下格式：

```JSON
{
  "code": 0,           // 整数，业务状态码
  "message": "success", // 字符串，状态描述
  "data": { ... }      // 可选，具体数据
}
```

- 成功时 `code` 为 `0`，`message` 通常为 `"success"`。

- 失败时 `code` 为非零，`message` 为错误描述，`data` 可能为 `null` 或包含额外错误信息。

### 4.3 错误对象

当请求失败时，响应中的 `data` 可能包含错误详情：

```JSON
{
  "code": 1001,
  "message": "Unsupported file format",
  "data": {
    "details": "Only JPEG and PNG are allowed"
  }
}
```

## 5. 设计原则详解

### 5.1 为什么使用异步任务模式？

- **处理时间不可控**：某些任务（如视频处理、AI 推理）可能耗时较长，同步接口会导致客户端连接超时或阻塞。

- **资源削峰填谷**：通过任务队列缓冲突发请求，保护后端服务。

- **状态持久化**：任务状态和结果持久化存储，客户端可随时查询，提高可靠性。

### 5.2 为什么统一提交入口？

- **简化客户端**：客户端只需知道一个提交 URL，无需为不同类型任务适配不同接口。

- **易于扩展**：新增任务类型只需在 `type` 字段中增加枚举值，并更新业务逻辑，接口本身无需修改。

### 5.3 为什么保留取消和删除接口？

- **取消**：允许用户中止不必要的任务，节省资源。

- **删除**：满足数据隐私需求（如 GDPR），允许用户彻底删除自己的数据。同时帮助系统清理过期数据。

### 5.4 如何保证接口稳定性？

- **版本控制**：URL 中包含主版本号（如 `/v1/`），未来不兼容更新时发布新版本。

- **预留扩展字段**：`data`、`config`、`result` 等均为对象，可随时增加新属性，客户端应忽略未知字段。

- **错误码规范**：使用业务错误码而非仅依赖 HTTP 状态码，便于客户端精细处理。

## 6. 使用示例

### 示例1：提交文本处理任务

bash

```JSON
curl -X POST https://api.example.com/v1/tasks \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "type": "text",
    "data": {
      "content": "这是一段需要处理的文本"
    },
    "config": {
      "language": "zh"
    }
  }'
```

响应：

```JSON
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "tsk_abc123",
    "type": "text",
    "status": "pending",
    "createdAt": "2025-03-15T10:00:00Z"
  }
}
```

### 示例2：轮询查询结果

```JSON
curl -X GET https://api.example.com/v1/tasks/tsk_abc123 \
  -H "Authorization: Bearer <token>"
```

响应（已完成）：

```JSON
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "tsk_abc123",
    "type": "text",
    "status": "completed",
    "createdAt": "2025-03-15T10:00:00Z",
    "completedAt": "2025-03-15T10:00:05Z",
    "result": {
      "sentiment": "positive",
      "keywords": ["示例", "文本"]
    }
  }
}
```

### 示例3：取消任务

```JSON
curl -X POST https://api.example.com/v1/tasks/tsk_abc123/cancel \
  -H "Authorization: Bearer <token>"
```

响应：

```JSON
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "tsk_abc123",
    "status": "cancelled",
    "updatedAt": "2025-03-15T10:01:00Z"
  }
}
```

### 示例4：删除任务

```JSON
curl -X DELETE https://api.example.com/v1/tasks/tsk_abc123 \
  -H "Authorization: Bearer <token>"
```

响应：`204 No Content`
